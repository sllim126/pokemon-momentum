Pokemon Momentum Data Flow
==========================

Recent Data/Serving Changes (March 2026)
----------------------------------------

1. Removed client dependency on full-universe preload for chart fetch
- Previous behavior:
  - chart selection could trigger a large `/universe` fetch when metadata cache missed
- Current behavior:
  - chart series requests derive `productId` and `subTypeName` directly from `currentKey`
  - no full-universe preload is required for normal chart loads
- Expected result:
  - lower browser-side network cost
  - faster tab/set/product transitions

2. Set Products server-side name fallback hardening
- `group_products` now detects placeholder metadata names (`Product <id>` / `ProductId <id>`).
- When placeholder is detected, server falls back to `product_signal_snapshot.productName` when available.
- Expected result:
  - fewer placeholder names in Set Products output even if metadata table contains stale placeholder rows

3. Client-side row normalization fallback hardening
- `dashboard_common.enrichRows()` now prioritizes non-placeholder payload names over cached metadata names.
- Expected result:
  - stale placeholder values in browser cache are less likely to overwrite valid API names

4. Loading UX for data-driven table flow
- Market Signals table now exposes an overlay spinner during active fetches.
- This is UI-only, but important for perceived data-flow status during larger queries.

5. More dashboard reads moved to snapshots
- `/health` now reads from a compact health snapshot instead of scanning history on request.
- `/series` now reads a prebuilt recent-series snapshot for normal chart windows.
- `/sparkline_batch` now reads prebuilt sparkline payloads instead of rebuilding tiny chart history live.
- `/universe` now reads from the product signal snapshot instead of scanning raw price history.
- the dashboard still applies client-side y-axis rules on top of `/series` so repriced windows
  can focus on the current visible price regime without losing SMA context
- snapshot-backed screeners now cover more of the first-load path, including the default mobile/home feeds.
- Expected result:
  - less request-time SQL work
  - faster first paint
  - less repeated work on phone/tablet refreshes

6. Search and screener workflow refinements
- header search now routes set-oriented selections into `Browse Set` instead of acting like a raw product jump
- `Good Buys` now filters for `recent_distinct_prices_30d >= 10`
- `Good Buys` now supports a client-side `Filter Sets` popout over the current result set
- `Breakouts` now honors the shared `Cards / Sealed` segment filter the same way as the other segmented tabs
- `Browse Set` now owns the set-scoped `Time to Buy` workflow through the `Set View` toggle
- `Browse Set` also has a client-side `Included Cards` filter that can narrow the loaded set rows by rarity and set-specific variants
- Expected result:
  - search drops the user into the correct set context more often
  - `Good Buys` is less likely to show thinly traded names by default and is easier to narrow to relevant sets

What this document is for
-------------------------
This file explains how data moves through the project from raw TCGCSV archives to the
dashboard screens. It is meant to answer:

- where the data comes from
- where it is stored
- which scripts are responsible for each stage
- which files/tables are considered intermediate vs final
- what the API and dashboards actually read from


High-level flow
---------------

1. Download raw daily archives from TCGCSV
2. Extract those archives to disk
3. Load category-specific price history into DuckDB
4. Refresh group and product metadata
5. Build joined/named exports for inspection
6. Build product-level and group-level signal snapshots
7. Export partitioned parquet history
8. Serve dashboards and API from DuckDB + parquet

Host-driven daily automation now runs that pipeline for both category `3` and category `85`
sequentially, so Japanese snapshots are rebuilt during the same nightly maintenance window
instead of depending on a separate manual pass.


Recent dashboard workflow additions
----------------------------------
The pipeline stages are unchanged, but the dashboard now uses the data with a few extra
workflow features:

- refresh-safe URL state
  - the selected market, product, tab, chart window, rail segment, selected set, and
    active group generation filter are stored in the URL
  - refreshes reopen the same research context

- persistent selected-item image preview
  - the selected product image now stays visible in the selected-product header / side context
  - clicking that image opens a small anchored preview without leaving the dashboard

- keyboard list stepping
  - the lower table can be navigated with Arrow Up / Arrow Down
  - this is a frontend convenience layer on top of the same product-selection flow

- new snapshot-driven screen: `Good Buys`
  - this uses the product snapshot plus rarity metadata to surface premium-rarity cards
    that have been trending down or pulling back
  - it now also requires recent distinct-price activity so thinly traded pullbacks are filtered out
  - the dashboard can now narrow those rows client-side to just the selected sets without rebuilding the snapshot

- new snapshot-driven companion page: `Set Explorer`
  - this page is intentionally lighter than the main dashboard
  - it is meant for browsing set cost and value concentration questions without needing
    to interpret chart signals first

- mobile-first dashboard workflow
  - selected product opens as a drilldown page on phone layouts
  - a mobile bottom nav coordinates `Home`, `Search`, `Tracked`, and `Indicators`
  - mobile home includes a `Singles / Sealed` segment toggle
  - pull-to-refresh refreshes dashboard state by calling the same snapshot-backed feeds

- search-to-set handoff
  - set and product search results can route directly into the `Browse Set` screen
  - the selected result is then resolved inside the set browser when possible

- uploaded set-logo support
  - local files from `/images/set logos` can now be synced into `/images/logos/<groupId>.<ext>`
  - Group Signals and Set Explorer both look for those synced groupId-based logo files
  - this keeps set imagery independent from individual product/card imagery

- stricter breakout screening
  - Breakouts no longer means "flat at the top of a range"
  - the screen now requires a real break above the prior 90-day high plus recent distinct-price activity

- lightweight tracking accounts
  - anonymous tags can still stay browser-local
  - signed-in users can now sync `favorite`, `watchlist`, `research`, and `buy list` tags
    through a separate tracking store
  - the header account menu is now the main entry point for sign-in, tracked items, and settings
  - the same menu now exposes public links to `Set Explorer` and `Sealed Deals`
  - `Report a Bug` submissions are stored separately with structured page-state context for later review

- paged picker loading
  - the dashboard `Product Picker` and `Set Picker` no longer rely on a full upfront catalog preload
  - both pickers now request paged results from `/product_picker` and `/groups`
  - this reduces first-load cost while preserving scrollable dropdown behavior

- removed `Signals` screen from dashboard UI
  - the broader product snapshot endpoint still exists, but the public dashboard no longer surfaces
    that tab because it was not proving useful enough in the current workflow


Core idea
---------
The long-term historical fact data is stored in two layers:

- DuckDB
  - working analytical database
  - used for loading, joins, metadata, and snapshot tables

- Parquet
  - partitioned historical fact store
  - used for large history reads when available

In practice:

- DuckDB is the working engine
- parquet is the durable historical read layer
- the API chooses parquet when a category exists there, otherwise it falls back to DuckDB

Stage 1: Raw download
---------------------
Script:

- [Download_new_day.py](/opt/pokemon-momentum/scripts/download/Download_new_day.py)

Purpose:

- download daily `.7z` archives from TCGCSV
- store them in `/app/data/raw`
- extract only newly downloaded archives into `/app/data/extracted`

Important detail:

- this is not category-specific
- one downloaded archive contains many category folders
- category `3` and category `85` both come from the same extracted day package

Result of this stage:

- raw archive files in `/app/data/raw`
- extracted folders like:
  - `/app/data/extracted/prices-YYYY-MM-DD.ppmd/...`


Stage 2: Historical price load
------------------------------
Script:

- [build_pokemon_prices_all_days.py](/opt/pokemon-momentum/scripts/extract/build_pokemon_prices_all_days.py)

Purpose:

- read extracted day folders
- read only the requested category subfolder
- parse each group price file
- bulk insert rows into DuckDB table `pokemon_prices`

Main storage target:

- DuckDB table: `pokemon_prices`

Important detail:

- this table is shared across categories
- category separation happens through the `categoryId` column

Optional export:

- category-specific CSV like:
  - `pokemon_prices_all_days.csv`
  - `pokemon_jp_prices_all_days.csv`

Those CSVs are mostly compatibility/debug artifacts now, not the preferred long-term store.

Result of this stage:

- DuckDB has historical fact rows for the selected category
- missing dates are added incrementally


Stage 3: Group metadata refresh
-------------------------------
Script:

- [export_pokemon_groups.py](/opt/pokemon-momentum/scripts/utilities/export_pokemon_groups.py)

Purpose:

- fetch all group/set metadata for the selected category from TCGCSV
- write category-specific CSV
- write category-specific DuckDB table

Outputs:

- CSV:
  - `pokemon_groups.csv`
  - `pokemon_jp_groups.csv`
- DuckDB tables:
  - `pokemon_groups`
  - `pokemon_jp_groups`

Result of this stage:

- set names, abbreviations, and metadata are available for joins and dashboard display


Stage 4: Product metadata refresh
---------------------------------
Script:

- [export_products_for_my_groups.py](/opt/pokemon-momentum/scripts/utilities/export_products_for_my_groups.py)

Purpose:

- look at which groups exist in `pokemon_prices` for the selected category
- fetch product metadata only for those groups
- by default, do this incrementally
- optionally, do a full refresh

Important detail:

- daily runs are incremental
- weekly runs force a full metadata refresh

Outputs:

- CSV:
  - `pokemon_products.csv`
  - `pokemon_jp_products.csv`
- DuckDB tables:
  - `pokemon_products`
  - `pokemon_jp_products`

Placeholder behavior:

- if price history contains a productId that metadata does not return, a placeholder row is created
- this keeps downstream joins from dropping priced products

Incremental refresh safeguard:

- incremental product refreshes now re-fetch any group whose cached metadata still contains
  placeholder names like `Product 586829`
- this prevents stale fallback rows from becoming permanent when tcgcsv later starts returning
  the missing product metadata
- dashboard/API metadata reads also prefer the category CSV when it is newer than the DuckDB
  file, which helps if a metadata refresh updated the CSV but the DuckDB table was not rebuilt yet

Result of this stage:

- product names, images, rarity, and number metadata exist for the selected category


Stage 5: Joined named history export
------------------------------------
Script:

- [join_prices_to_names.py](/opt/pokemon-momentum/scripts/utilities/join_prices_to_names.py)

Purpose:

- join raw price history to set and product names
- create a human-readable export for inspection and debugging

Outputs:

- CSV:
  - `pokemon_prices_named.csv`
  - `pokemon_jp_prices_named.csv`
- DuckDB tables:
  - `pokemon_prices_named`
  - `pokemon_jp_prices_named`

Result of this stage:

- there is an easy-to-read historical export with both numeric IDs and human names


Stage 6: Product signal snapshot
--------------------------------
Script:

- [build_product_signal_snapshot.py](/opt/pokemon-momentum/scripts/indicators/build_product_signal_snapshot.py)

Purpose:

- take the latest available row for each product/subtype
- compute moving-average context and rate-of-change values
- classify product type
- compute a product-level trend score

Inputs:

- `pokemon_prices`
- category-specific groups table
- category-specific products table

Outputs:

- CSV:
  - `pokemon_product_signal_snapshot.csv`
  - `pokemon_jp_product_signal_snapshot.csv`
- DuckDB tables:
  - `pokemon_product_signal_snapshot`
  - `pokemon_jp_product_signal_snapshot`

Important fields:

- `roc_7d_pct`
- `roc_30d_pct`
- `roc_90d_pct`
- `price_vs_sma30_pct`
- `price_vs_sma90_pct`
- `breakout_90d_flag`
- `acceleration_7d_vs_30d`
- `trend_score`
- `productClass`
- `productKind`

Result of this stage:

- the API can serve momentum-style product screens without recomputing every metric on each request


Stage 7: Group signal snapshot
------------------------------
Script:

- [build_group_signal_snapshot.py](/opt/pokemon-momentum/scripts/indicators/build_group_signal_snapshot.py)

Purpose:

- roll the product snapshot up to the set/group level
- measure breadth and participation across a whole set

Inputs:

- category-specific product signal snapshot

Outputs:

- CSV:
  - `pokemon_group_signal_snapshot.csv`
  - `pokemon_jp_group_signal_snapshot.csv`
- DuckDB tables:
  - `pokemon_group_signal_snapshot`
  - `pokemon_jp_group_signal_snapshot`

Important fields:

- `item_count`
- `card_count`
- `sealed_count`
- `avg_30d_pct`
- `avg_90d_pct`
- `pct_above_sma30`
- `pct_at_90d_high`
- `breadth_score`

Result of this stage:

- the API can rank whole sets, not just individual cards/products

Recent API-side addition:

- the dashboard now derives a broad generation/era label for each set when serving
  `Group Signals`
- this label is built from group metadata such as set name, abbreviation, and
  publication date so the UI can filter set breadth screens by era
- Japanese Mega-era overrides now cover multiple modern Mega prefixes so sets like `M2`,
  `M2a`, `M3`, `MBD`, `MBG`, and `MP1` stay in the `MEG` bucket instead of falling into generic SV


Stage 8: Parquet export
-----------------------
Script:

- [export_parquet.py](/opt/pokemon-momentum/scripts/utilities/export_parquet.py)

Purpose:

- export each date in the price fact table into a parquet partition

Output location:

- `/app/data/parquet/date=YYYY-MM-DD/data.parquet`

Important detail:

- parquet is partitioned by date, not by category
- if a given date contains rows for categories `3` and `85`, both will be present in that date's parquet file

Current API behavior:

- prefer parquet for history reads when the category exists there
- fall back to DuckDB if parquet does not yet contain that category

Result of this stage:

- large historical scans can use parquet rather than only the DuckDB fact table

----------------------------

Purpose:

- confirm required outputs exist

API layer
---------
Main API:

- [api.py](/opt/pokemon-momentum/scripts/dashboards/api.py)

What it does:

- serves dashboard HTML files
- exposes JSON endpoints for universe, groups, series, movers, signals, and set views
- picks the best history source for each category
- exposes newer workflow endpoints like:
  - `/good_buys`
  - `group_signals?generation=...`
  - `/group_series?groupId=...`
  - `/set_baskets`
  - `/tracking/...`

Set history endpoint:

- `/group_series`
  - builds an on-demand set-level time series from the raw daily history
  - returns:
    - equal-weight set index
    - daily percent of items above SMA30
    - active item count
    - average and median set price
  - this is what lets `Group Signals` drive the main chart without forcing an immediate drilldown to one product

Set explorer endpoint:

- `/set_baskets`
  - builds a condensed set-comparison table from the latest product signal snapshot
  - returns:
    - tracked set cost
    - card count
    - top hit name and price
    - top hit share of total set value
    - top 3 share of total set value
    - depth score
  - this powers the separate `/set-explorer` page
  - it is designed for "poke around in the set data" questions rather than live chart analysis

Tracking endpoints:

- `/tracking/session`
  - creates or resumes a lightweight tracking account with username + PIN

- `/tracking/tags`
  - reads and writes saved workflow tags for one signed-in user

- `/tracking/tags/merge`
  - pushes locally-created tags into the saved account after sign-in

- `/tracking/change_pin`
  - updates the lightweight tracking account PIN

- `/tracking/account`
  - deletes the tracking account, tags, and active sessions

Important helpers:

- `prices_from(category_id)`
  - chooses parquet if available for that category
  - otherwise uses DuckDB table `pokemon_prices`

- `products_from(category_id)`
  - chooses category-specific metadata table or CSV fallback

- `groups_from(category_id)`
  - chooses category-specific groups table or CSV fallback

- `product_signal_from(category_id)`
  - chooses category-specific product signal table or CSV fallback

- `group_signal_from(category_id)`
  - chooses category-specific group signal table or CSV fallback

Result:

- dashboard code does not need to know where the storage currently lives
- it only needs to pass `category_id`

Source-of-truth guidance
------------------------
If you are trying to decide where something "really lives", use this mental model:

- Raw upstream source:
  - TCGCSV daily archives

- Working historical fact table:
  - DuckDB `pokemon_prices`

- Durable/scalable historical read layer:
  - parquet partitions in `/app/data/parquet`

- Metadata:
  - category-specific groups/products tables

- Latest analytical state:
  - category-specific product/group signal snapshot tables

- Tracking state:
  - SQLite `tracking.sqlite3`
  - intentionally separate from the price-history pipeline

- UI/API:
  - reads from parquet when it can
  - reads from DuckDB when parquet does not yet cover that category or date


Typical daily success path
--------------------------
On a normal successful daily run, the expected sequence is:

1. new archive is downloaded if available
2. new day is extracted if needed
3. missing price rows are inserted into `pokemon_prices`
4. group metadata is refreshed
5. product metadata is refreshed
6. joined named export is rebuilt
7. product signal snapshot is rebuilt
8. group signal snapshot is rebuilt
9. new parquet partitions are exported
10. API serves the new latest date
11. dashboards can immediately use the refreshed snapshot-driven screens, including
    `Good Buys`, generation-filtered `Group Signals`, and the lighter `Set Explorer`


Common failure patterns
-----------------------

If the dashboard latest date does not advance:

- parquet probably has not been exported yet
- or the category is missing from parquet and the API is still preferring it incorrectly

If a category loads in snapshots but not in history endpoints:

- snapshot tables exist
- but parquet for that category may be missing
- the API should then fall back to DuckDB

If set/product names are missing:

- metadata tables may not have refreshed
- placeholder rows may be in use

If the pipeline hangs during metadata:

- tcgcsv product/group endpoints may be slow or flaky
- incremental metadata refresh reduces this risk for daily runs


Recommended long-term architecture
----------------------------------
The intended long-term shape of the system is:

- TCGCSV archive download
- extracted day folders
- DuckDB ingest / metadata / snapshots
- parquet as historical fact read layer
- FastAPI as the query + presentation layer
- dashboards as research/public views on top

That means the project is not choosing between DuckDB or parquet.
It is using both:

- DuckDB for loading, joins, snapshots, and API-friendly analytics
- parquet for large historical storage and scalable reads
