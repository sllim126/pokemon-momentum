Store Pricing Ops
-----------------
The store pricing workflow now has three admin pages that work together:

- `/pricing-upload`
  - upload the latest Squarespace product export after you add or edit products
- `/supplier-pricing`
  - upload or OCR supplier screenshots and save reviewed JPY quote rows by SKU
- `/supplier-profitability`
  - compare latest supplier costs against market/store pricing with editable supplier-side and buyer-side assumptions

Daily price sync inputs:

- Squarespace export:
  - `/opt/pokemon-momentum/products_Apr-09_04-31-18PM.csv`
- Supplier quotes:
  - `/opt/pokemon-momentum/data/supplier_quotes.csv`
- Price rules:
  - `/opt/pokemon-momentum/data/store_price_rules.csv`
- Generated daily targets:
  - `/opt/pokemon-momentum/data/market_prices_latest.csv`

Important pricing behavior:

- the target builder now uses `max(market_target, profit_floor)` for SKUs with supplier quotes
- `profit_floor` is derived from supplier cost plus landed-cost assumptions
- rows without supplier quotes remain market-based
- `target_source` in `market_prices_latest.csv` shows whether a row came from `market` or `profit_floor`

Recommended operator flow when adding a new SKU:

1. Add the product in Squarespace with a unique SKU.
2. Add or update its pricing rule in `data/store_price_rules.csv`.
3. Upload the refreshed Squarespace export on `/pricing-upload`.
4. Save the latest supplier quote on `/supplier-pricing` if you want profit-floor protection.
5. Review `/supplier-profitability` before allowing auto-pricing to move the SKU.

Storefront linking (`Poke6s` button)
------------------------------------

The dashboard can now show a `Poke6s` button beside `TCGplayer`, but only when the
current item is actually purchasable on the storefront.

Files involved:

- Squarespace export:
  - `products_Apr-09_04-31-18PM.csv`
- Store/TCGplayer mapping:
  - `data/squarespace_tcgplayer_mapping.csv`

Current link rules:

- the storefront button only appears when the latest Squarespace export says:
  - `Visible = Yes`
  - `Stock > 0` or `Stock = Unlimited`
- Japanese and English singles can resolve directly when:
  - Squarespace `SKU == TCGplayer product ID`
- sealed/store products resolve through:
  - `data/squarespace_tcgplayer_mapping.csv`
- one store SKU can map to multiple TCGplayer product ids by separating ids with `|`
  or `,` in `tcgplayer_product_id`

Recommended operator flow when a dashboard item should link to the storefront:

1. Confirm the Squarespace product is visible and in stock in the latest export.
2. If it is a single, confirm the store SKU matches the TCGplayer product id.
3. If it is a sealed/store product, add or update its row in `data/squarespace_tcgplayer_mapping.csv`.
4. Restart the app if the dashboard is still serving an older resolver path.

If the `Poke6s` button appears when it should not:

1. Check the selected `productId` in the dashboard URL or search payload.
2. Confirm whether that exact `productId` is mapped in `data/squarespace_tcgplayer_mapping.csv`.
3. Watch out for near-duplicate TCGplayer products:
  - sleeved pack vs unsleeved pack
  - art bundle vs standard pack bundle
  - ETB vs Pokemon Center ETB

If the `Poke6s` button is missing when it should appear:

1. Check the latest Squarespace export row for:
  - `Visible`
  - `Stock`
  - `Product URL`
2. Confirm the dashboard category matches the actual item market:
  - English -> category `3`
  - Japanese -> category `85`
3. For sealed products, confirm the exact TCGplayer product id is mapped to the store SKU.

What each page is expected to do
--------------------------------

`/pricing-upload`

- Purpose:
  - replace the local Squarespace export with a fresh one after store edits
- Inputs:
  - a valid Squarespace product CSV export
- Outputs:
  - archive copy in `data/archive/squarespace_exports/`
  - canonical working export in `products_Apr-09_04-31-18PM.csv`
  - coverage preview showing which SKUs have usable pricing rules
- Debug questions:
  - if a SKU is missing from automation, did it appear in the uploaded export?
  - if the export was accepted but no preview row appears, check `unmatched_rules`

`/supplier-pricing`

- Purpose:
  - convert supplier screenshots into structured SKU-linked quote rows
- Inputs:
  - OCR-reviewed rows containing supplier item name and JPY cost
- Outputs:
  - appended historical quote rows in `data/supplier_quotes.csv`
- Debug questions:
  - if OCR rows have blank SKU, check `data/supplier_name_mapping.csv`
  - if a quote was saved but not reflected later, confirm the SKU was populated on save

`/supplier-profitability`

- Purpose:
  - answer whether one unit is profitable under current supplier-side and buyer-side assumptions
- Inputs:
  - latest supplier quote per SKU
  - generated market/store target data
  - operator assumptions from the page
- Outputs:
  - per-unit rows with landed cost, break-even, target-margin price, and decision status
- Important interpretation:
  - results are per unit, not whole-order totals
  - `Estimate from Order` allocates whole-order shipping down to one box
  - `Manual per Item` ignores order shipping and uses only the manual inbound value

Generated target file fields
----------------------------

`data/market_prices_latest.csv` now includes:

- `market_price`
  - latest market price pulled from the signal snapshot
- `target_price`
  - final price the sync script will try to push to Squarespace
- `profit_floor_price`
  - minimum viable target derived from supplier economics, blank when no supplier quote exists
- `supplier_cost_jpy`
  - raw supplier quote used for floor calculation
- `target_source`
  - `market` when market logic won
  - `profit_floor` when supplier economics overrode the market target

If pricing looks wrong:

1. Check whether the SKU appears in `products_Apr-09_04-31-18PM.csv`.
2. Check its rule row in `data/store_price_rules.csv`.
3. Check its latest supplier quote in `data/supplier_quotes.csv`.
4. Rebuild targets:

```bash
cd /opt/pokemon-momentum
python3 scripts/build_store_price_targets.py
```

5. Inspect the resulting row in `data/market_prices_latest.csv`.

Pricing coverage audit:

```bash
cd /opt/pokemon-momentum
python3 scripts/utilities/audit_store_pricing_coverage.py
```

- `automated`
  - visible SKU is covered by the generated pricing target file
- `manual_expected`
  - visible SKU is intentionally manual, such as custom or print-on-demand items
- `hold_expected`
  - visible SKU is intentionally excluded until release or usable market data exists
- `missing_unexpected`
  - visible SKU is missing a rule or failed to generate a target and needs attention

Daily automation:

- the host-driven daily update now runs this audit once after the category pipelines finish
- if the audit finds `missing_unexpected` SKUs, the daily update log records a warning without failing the whole refresh

Normal daily operation
----------------------
The system is intended to run automatically.

Daily timer:

- `pokemon-momentum-daily-update.timer`

Daily service:

- `pokemon-momentum-daily-update.service`

Weekly full metadata refresh:

- `pokemon-momentum-weekly-refresh.timer`
- `pokemon-momentum-weekly-refresh.service`

Weekly placeholder cleanup refresh:

- `pokemon-momentum-weekly-placeholder-refresh.timer`
- `pokemon-momentum-weekly-placeholder-refresh.service`

Normal expectation:

- the daily run appends new price data
- refreshes metadata incrementally
- rebuilds snapshots
- exports new parquet partitions
- processes both category `3` (Pokemon) and category `85` (Pokemon Japanese) in sequence
- restarts the app container afterward
- the weekly full metadata refresh runs Sunday at `11:00 UTC`
- the weekly placeholder cleanup runs Monday at `11:00 UTC`, 24 hours later
- the placeholder cleanup only reruns product metadata + product signal snapshot so stale
  `Product <id>` rows get another chance to resolve

If the pipeline appears stuck
----------------------------
Symptoms:

- `daily_update.log` stops advancing for a long time
- memory pressure spikes

Where this usually happens:

- metadata fetches from tcgcsv
- large price-load step

Checks:

```bash
tail -f /opt/pokemon-momentum/logs/daily_update.log
sudo docker stats
free -h
```

Notes:

- the daily path is now tuned to use a modest worker count
- one-time backfills are intentionally heavier
- metadata fetches can still be slow because they depend on tcgcsv responses

Recommended action:

- give the run time to finish if it is still advancing
- only intervene if logs are fully stationary and the process clearly failed or was OOM-killed


If DuckDB reports a file lock error
-----------------------------------
Symptoms:

- error about conflicting lock on `prices_db.duckdb`

Cause:

- the app container is running and reading from the DB while a write process tries to start

Normal design:

- the host wrapper stops the app before pipeline writes

Manual recovery:

```bash
cd /opt/pokemon-momentum
sudo docker-compose stop pokemon-momentum
sudo docker-compose run --rm pokemon-momentum bash -lc 'python scripts/pipeline/run_daily_update.py --skip-download --workers 4'
sudo docker-compose up -d pokemon-momentum
```

If dashboards load but feel wrong
---------------------------------
Symptoms:

- empty rails
- chart not updating
- market switch empties the page
- set browser does not load products

Checks:

1. API endpoints first

```bash
curl 'http://localhost:8001/categories'
curl 'http://localhost:8001/health?category_id=3'
curl 'http://localhost:8001/health?category_id=85'
```

2. restart app

```bash
cd /opt/pokemon-momentum
sudo docker-compose restart pokemon-momentum
```

3. hard refresh browser

- `Ctrl+Shift+R`


Standard recovery sequence
--------------------------
If something is wrong and you want the safest basic recovery path:

1. check `/health`
2. check `daily_update.log`
3. restart the app container
4. rerun the daily service manually
5. if latest date is still stale, run parquet export
6. if metadata is wrong, rerun metadata refresh
