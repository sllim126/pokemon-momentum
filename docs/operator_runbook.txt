Pokemon Momentum Operator Runbook
=================================

Recent Operational Notes (March 2026)
-------------------------------------

These checks were added after a dashboard performance and UX pass.

1. Verify no unintended full-universe browser fetch
- In browser DevTools Network, normal tab/set switching should not repeatedly trigger large
  `universe?limit=...` calls.
- If seen, hard refresh first (`Ctrl+Shift+R`) to clear stale frontend code.

2. Verify Market Signals loading feedback
- During slow screener calls, table area should show a spinner overlay with loading text.
- If table appears frozen without spinner, check that latest `dashboard.html` is loaded.

3. Placeholder name triage (`Product <id>`)
- First check whether API now resolves names:
  - `GET /group_products?groupId=<id>&limit=...`
- If placeholder names still appear broadly, run metadata+snapshot refresh:

```bash
cd /opt/pokemon-momentum
sudo docker-compose exec -T pokemon-momentum python scripts/utilities/export_products_for_my_groups.py --category-id 3
sudo docker-compose exec -T pokemon-momentum python scripts/indicators/build_product_signal_snapshot.py --category-id 3
sudo docker-compose restart pokemon-momentum
```

- Expected result:
  - metadata placeholders are re-evaluated
  - product snapshot fallback has fresher names available

4. Post-change smoke test checklist
- `/dashboard` loads
- switching Market Signals tabs auto-loads first row into chart
- Set Products loads without generic `Product <id>` for known named sets
- header search set results open directly into `Browse Set`
- header search product results can open `Browse Set` with the matching item selected
- no unexpected large universe preloads on routine interactions
- mobile bottom nav works:
  - `Home`
  - `Search`
  - `Tracked`
  - `Indicators`
- mobile pull-to-refresh refreshes the current view without a full browser reload
- clicking a product image opens the anchored in-page preview and clicking it again closes it

It answers:

- what the common recovery steps are

Normal daily operation
----------------------
The system is intended to run automatically.

Daily timer:

- `pokemon-momentum-daily-update.timer`

Daily service:

- `pokemon-momentum-daily-update.service`

Weekly full metadata refresh:

- `pokemon-momentum-weekly-refresh.timer`
- `pokemon-momentum-weekly-refresh.service`

Weekly placeholder cleanup refresh:

- `pokemon-momentum-weekly-placeholder-refresh.timer`
- `pokemon-momentum-weekly-placeholder-refresh.service`

Normal expectation:

- the daily run appends new price data
- refreshes metadata incrementally
- rebuilds snapshots
- exports new parquet partitions
- processes both category `3` (Pokemon) and category `85` (Pokemon Japanese) in sequence
- restarts the app container afterward
- the weekly full metadata refresh runs Sunday at `11:00 UTC`
- the weekly placeholder cleanup runs Monday at `11:00 UTC`, 24 hours later
- the placeholder cleanup only reruns product metadata + product signal snapshot so stale
  `Product <id>` rows get another chance to resolve


Quick health checklist
----------------------
If you just want to know whether the system is healthy, check these first.

1. API responds

```bash
curl http://localhost:8001/health
curl 'http://localhost:8001/health?category_id=85'
```

Healthy result:

- JSON returns successfully
- `rows` is nonzero
- `latest` is a real date
- `source` is either `parquet` or `duckdb`

2. Main dashboards load

- `/dashboard`
- `/embed`

Healthy result:

- page renders
- left rail loads
- chart loads
- market toggle works
- refresh preserves current dashboard state
- long signal/set lists scroll inside the lower panel
- `set-explorer` loads and the set table populates
- tracking sign-in works and tags persist after refresh
- `Signals` tab is no longer present in the dashboard UI
- `Good Buys` supports a `Filter Sets` popout so review can be narrowed to the specific sets you want to inspect
- on phone/tablet widths:
  - mobile home shows `Singles / Sealed`
  - selected-product drilldown hides the indicator tabs below the chart

3. systemd timers are active

```bash
systemctl list-timers | grep pokemon-momentum
```

Healthy result:

- daily timer listed
- weekly timer listed
- weekly placeholder timer listed

------------------
Symptoms:

- `/health` does not respond
- dashboard page does not load

Recovery:

```bash
cd /opt/pokemon-momentum
sudo docker-compose ps
sudo docker-compose logs --tail=200 pokemon-momentum
sudo docker-compose restart pokemon-momentum
```

If restart is not enough:

```bash
cd /opt/pokemon-momentum
sudo docker-compose up -d --build
```


If the latest date is stale
---------------------------
Symptoms:

- dashboard shows an older latest date than expected
- `/health` shows an older latest date

Likely causes:

- daily pipeline has not finished
- parquet export did not run
- app restarted before the export stage completed

Checks:

```bash
tail -f /opt/pokemon-momentum/logs/daily_update.log
curl http://localhost:8001/health
```

What you want to see in the log:

- `Export parquet partitions`
- successful completion after that step

Manual recovery:

```bash
cd /opt/pokemon-momentum
sudo docker-compose run --rm pokemon-momentum bash -lc 'python scripts/utilities/export_parquet.py'
sudo docker-compose restart pokemon-momentum
curl http://localhost:8001/health
```


If the daily service fails immediately
-------------------------------------
Symptoms:

- `systemctl start pokemon-momentum-daily-update.service` fails fast

Checks:

```bash
sudo systemctl status pokemon-momentum-daily-update.service
sudo journalctl -xeu pokemon-momentum-daily-update.service
```

Common causes:

- bad service PATH
- script error in the pipeline
- Docker not available

Recovery:

- inspect the failing step in the journal
- fix the script/config problem
- rerun the service manually


If the pipeline appears stuck
----------------------------
Symptoms:

- `daily_update.log` stops advancing for a long time
- memory pressure spikes

Where this usually happens:

- metadata fetches from tcgcsv
- large price-load step

Checks:

```bash
tail -f /opt/pokemon-momentum/logs/daily_update.log
sudo docker stats
free -h
```

Notes:

- the daily path is now tuned to use a modest worker count
- one-time backfills are intentionally heavier
- metadata fetches can still be slow because they depend on tcgcsv responses

Recommended action:

- give the run time to finish if it is still advancing
- only intervene if logs are fully stationary and the process clearly failed or was OOM-killed


If DuckDB reports a file lock error
-----------------------------------
Symptoms:

- error about conflicting lock on `prices_db.duckdb`

Cause:

- the app container is running and reading from the DB while a write process tries to start

Normal design:

- the host wrapper stops the app before pipeline writes

Manual recovery:

```bash
cd /opt/pokemon-momentum
sudo docker-compose stop pokemon-momentum
sudo docker-compose run --rm pokemon-momentum bash -lc 'python scripts/pipeline/run_daily_update.py --skip-download --workers 4'
sudo docker-compose up -d pokemon-momentum
```

Standard recovery sequence
--------------------------
If something is wrong and you want the safest basic recovery path:

1. check `/health`
2. check `daily_update.log`
3. restart the app container
4. rerun the daily service manually
5. if latest date is still stale, run parquet export
6. if metadata is wrong, rerun metadata refresh
